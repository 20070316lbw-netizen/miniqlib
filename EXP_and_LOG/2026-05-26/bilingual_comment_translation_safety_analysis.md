## 🛑 深度剖析：为什么改注释会导致大型项目编译或运行崩溃？

在 Python 的世界里，很多人以为注释（`#` 和 `"""`）是不会被解释器执行的，所以可以随意修改。然而在集成了 C++ 和 Cython 的量化项目中，这种认知是极其致命的，原因如下：

### 1. Windows + MSVC + Cython 的“编码大天坑”（引发 C4819 编译崩溃）
* **原理**：Windows 系统的默认区域编码通常是 GBK（ANSI）。当 `uv` 或 `pip` 在本地使用 Visual Studio 的 MSVC 编译器编译 Cython（`.pyx`）或由其生成的 C/C++ 源文件时，MSVC 默认使用 GBK 格式去解码。
* **致命后果**：如果我们在 `.pyx` 或参与编译的 `.py` 文件注释里写了中文（UTF-8），MSVC 在读取这些中文字符时，会因为字节错乱直接抛出 **`compiler error C4819: 该文件包含不能在当前代码页中表示的字符`**。严重时，甚至会因为字节错位破坏了相邻的代码语法结构，导致整个 C++ 编译阶段彻底崩溃！
* **防范**：凡是参与 Cython 预编译的源文件，其内的注释绝对不能出现非 ASCII 字符（如中文）。

### 2. Cython 的“魔术编译注释”（Magic Comments & Compiler Directives）
* **原理**：Cython 依靠文件头部的特定注释来接收编译指令。例如：
  ```python
  # cython: language_level=3
  # distutils: language = c++
  ```
* **致命后果**：如果翻译软件或者智能助手不小心将这些魔术注释翻译成中文（如 `# 赛通: 语言级别=3`），或者在翻译过程中破坏了其严格的冒号和空格排版，Cython 编译器将无法识别编译目标，直接报 `SyntaxError` 拒绝编译。

### 3. 自动化测试中的 Doctest 污染（引发单元测试大面积报错）
* **原理**：许多高质量的开源项目（如 Qlib 核心组件）在函数的 `docstring` 中直接嵌入了交互式测试用例（Doctest）。例如：
  ```python
  """
  Example:
  >>> import qlib
  >>> qlib.init()
  """
  ```
  `pytest` 在运行时会默认提取并执行这些 docstring 里的代码来做单元测试。
* **致命后果**：如果在汉化 Docstring 时，不小心将 `>>>` 后面的 Python 测试代码进行了汉化，或者破坏了严格的缩进（Doctest 对空格缩进的要求极其苛刻），会导致自动化测试大面积崩溃，持续集成（CI/CD）直接中断。

### 4. 字符串反射与元数据提取崩溃
* **原理**：有些库的 `__init__.py` 或插件加载器，会在运行时使用 `sys` 或 `inspect` 模块去反射解析模块顶部的 Docstring，从而动态提取版本、作者、甚至通过 Docstring 的特定标记来加载插件。
* **致命后果**：修改了这些魔术 docstring，会导致运行时反射找不到对应的 tag，使得插件或核心模块加载直接失败。

## 🚧 翻译区域安全性普查划分

为了确保万无一失，我们将整个工作区的代码文件划分为三级安全梯度：

### 🔥 红色危险区（绝对严禁触碰，否则 100% 崩盘）
1. **所有 `.pyx`、`.pxd` 文件**：这些是 Cython 源码和头文件。它们必须直接送进 C++ 编译器，**绝对严禁更改其中任何注释**！
2. **构建配置文件（`setup.py`、`setup_cfg`、`pyproject.toml`）**：这些文件控制着整个编译链，任何由于中文字符带来的编码混乱都会导致 `uv sync` 构建虚拟环境失败。
3. **`zipline-reloaded/` 的 `src/` 目录**：Zipline 包含了大量的 C 语言扩展绑定，其编译链路极其脆弱，不能有任何多字节字符的干扰。
4. **`qlib/qlib/core/` 目录**：这是 Qlib 的底层时序数据中心 C 语言绑定接口层。

### 🟡 黄色敏感区（中度风险，在严格工程监督下可微调）
1. **`qlib/qlib/contrib/` 目录**：包含大量用户贡献的模型（如 `lightgbm`、`xgboost` 等）和回测策略。这些是纯 Python 文件，不参与 C 编译，但是在 Docstring 中含有大量的公式（LaTeX 格式）和 Doctest。
   * *策略*：如果翻译，**只能翻译其中非 LaTeX、非 Doctest 的纯文本解释部分**，且必须保持文件的 `UTF-8` 编码格式。
2. **`mini_qlib/data/ops.py` & `mini_qlib/fetcher/`**：我们自己写的底层数据处理操作和 SEC EDGAR 抓取器。
   * *策略*：已完美配置中英双语注释，未来新加代码也需严格按照双语对照模板编写。

### 🟢 绿色安全区（零风险，高度推荐翻译以极大辅助学习）
这些是高层应用代码、说明文档与 Notebook。它们绝对不参与任何底层编译，修改它们就如同修改纯文本，风险为零，回报极高！

## 📜 绿色候选区翻译无损工程规范

对于任何未来的绿色候选区翻译，我们必须强制遵守以下四项防崩溃纪律：

1. **🔒 Doctest 绝对锁死**：凡是遇到 `>>>` 开头的代码测试行，以及紧随其后的输出行，**严禁做任何字符、空格、缩进的修改**。
2. **📝 LaTeX 公式绝对保持原样**：凡是在 docstring 中被 `$` 或 `$$` 包裹的数学公式，必须一字不差地保留，只翻译前后的解释文字。
3. **💾 显式 UTF-8 固化**：修改过的文件在保存时，编辑器必须显式选择 `UTF-8` 编码格式（Windows 下严禁存为 UTF-8 with BOM 或 ANSI 格式）。
4. **🧪 灰度验证原则**：每翻译完一个应用层脚本的注释，必须立即在根目录下运行 `uv run python <translated_script>.py`，只有跑通且输出正常，才能进行 `git commit`。
